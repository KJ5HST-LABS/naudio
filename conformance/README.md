# net-audio Protocol Conformance Suite

Language-neutral **golden-vector** conformance fixtures for the [audio streaming
protocol v1](../docs/audio-streaming-protocol-v1.md). A conformance test
loads the vectors and validates encode + decode against them. Passing the
language-neutral vectors against an independently computed known-answer set is
the proof that an implementation matches the spec — the gate every reference
client (C, C++, Python, and any other binding) must also pass.

## Why these vectors are trustworthy

The vectors are a **known-answer test, not a round-trip-against-self.** The
expected bytes — including every CRC32 — are produced **independently** by
`tools/gen_vectors.py` (using Python's `zlib.crc32`), *not* by the
implementation under test. So when an implementation reproduces
`frame-heartbeat-seq5`'s CRC `0x156994f7`, that is agreement with a value
computed by a third party, not a round-trip against its own encoder.

## Regenerating

`vectors.ini` is committed — it is the **source of truth** the conformance test
reads. It is generated by `tools/gen_vectors.py`, which computes every expected
value (including the CRCs) independently of the implementation under test:

```
python3 conformance/tools/gen_vectors.py conformance/vectors/vectors.ini
```

Only regenerate when the spec's frozen contract intentionally changes — and then
expect the conformance test to need review. The byte-identity tell for a correctly
received stream is `first_frame_hex=68c569c66ac76bc8`; if a regenerated `vectors.ini`
ever changes the frame bytes, that hex will differ — a real wire-bug surface, not a
cosmetic drift.

## File format (`vectors.ini`)

A minimal INI: `# comment` lines, blank lines ignored, `[vector-name]` sections,
`key = value` lines. Values are trimmed. `*Hex` fields are lowercase hex strings
(empty string = empty byte array). This format is deliberately stdlib-parseable
in any language (no JSON/YAML dependency — important for offline builds and for
the C/Python clients).

Each vector has a `kind`; the loader dispatches on it. Field schema by kind:

### `kind = crc32`
- `inputHex` — bytes to checksum.
- `expected` — 8-hex-digit CRC-32/ISO-HDLC. Assert `CRC32(inputHex) == expected`.

### `kind = frame`
A full `0xAF01` frame (header + payload + CRC), timestamp pinned (see below).
- `expectDecode` — `accept` or `reject`.
- `frameHex` — the exact wire bytes.
- For `accept`: `type`, `sequence`, `flags`, `timestamp`, `payloadHex` are the
  fields. The test asserts **decode** (`deserialize(frameHex)` yields those
  fields) **and encode** (building a packet from those fields and serializing
  yields `frameHex` exactly).
- For `reject`: `deserialize(frameHex)` must yield "no packet" (null/None), not
  throw.

### `kind = control`
A `ControlMessage` payload (the bytes inside a `CONTROL` frame).
- `controlType` — the control-type byte (e.g. `0x04` AUDIO_CONFIG).
- `expectedPayloadHex` — the serialized control payload (`type byte + data`).
- Type-specific fields: AUDIO_CONFIG → `sampleRate/bitsPerSample/channels/
  frameDurationMs/bufferTargetMs/bufferMinMs/bufferMaxMs`; NACK/CONTROL_ACK →
  `sequence`; CONNECT_REJECT → `rejectReason/message`.

### `kind = fec_encode`
- `blockSize`, `startSequence`, `payloadsHex` (comma-separated payloads).
- `expectedXorHex`, `expectedParityPayloadHex` (`startSeq i32 + blockSize u8 +
  xorData`). Assert the encoder's parity payload equals `expectedParityPayloadHex`.

### `kind = fec_recover`
- `blockSize`, `startSequence`, `presentHex` (`seq:payload,seq:payload`),
  `parityPayloadHex`, `missingSequence`, `expectedRecoveredHex`. Drive the
  decoder with the present packets + a `null` for the missing sequence + the
  parity; assert the recovered packet's payload equals `expectedRecoveredHex`.

### `kind = reorder`
- `windowSize`, `maxHoldMs`, `inputSeqs` (comma-separated), `expectedEmitted`
  (comma-separated; `GAP` marks an emitted null), `expectedReordered`,
  `expectedDroppedLate`, `expectedGaps`.

### `kind = jitter`
- `minMs`, `maxMs`, `multiplier`, `packets` (count), `expectedTargetMs`,
  `expectedPacketCount`.

## Notes

- **Timestamps are pinned to 0** in frame vectors. The frame timestamp is sampled
  at packet creation (§3.5 of the spec) and its clock origin is
  implementation-defined (e.g. a monotonic process clock, or Unix epoch), so a
  byte-exact vector *must* pin it. Conformance never asserts timestamp equality
  for live (un-pinned) frames.
- **Jitter vectors are structural only** (`empty → -1`, `first packet → minMs`).
  These are reproducible through the public API in any language without injecting
  arrival times. The full EMA/ramp arithmetic (gain 1/16, multiplier, asymmetric
  ramp) is specified normatively in the spec §8.1 and covered by each
  implementation's own unit tests, but is **not** cross-validated here because
  arrival-time injection is not a guaranteed-symmetric public API across
  implementations. A future revision may add a neutral arrival-injection hook and
  promote the exact jitter math into the golden vectors.
- The suite validates the **wire contract and the deterministic pure functions**
  (frame codec, CRC, control messages, FEC parity bytes + recovery, reorder
  ordering). Live socket interop (the handshake over real TCP/UDP loopback) is
  exercised separately by the networking tests (`naudio_c_net_smoke` and the
  `tests/net/` suite), not by these golden vectors.
